And we have also received a significant amount of help from the Net community but we could definitely use more help!

Building the development version of Cygwin

The code corresponding to cygwin1.dll is mostly in the winsup directory with additional code coming from newlib/libc, newlib/libm, libiberty, and gcc. Any changes you make should ideally be made to the current development sources. This makes it much easier to integrate your changes once they're completed and tested. Note the development snapshots should compile, but they should always be assumed to be unstable and contain untested, experimental code.

To build the Cygwin library, start by building the comp-tools sources with the last release, substituting in the newlib and winsup sources from the snapshot location. Minimal build instructions are included in the FAQ. If your object directory is /c/obj, the Cygwin dll you will build should be /c/obj/winsup/new-cygwinxx.dll where xx represents the current API_MAJOR version number.

Once you have a built comp-tools tree, if you make further changes to the winsup sources, you can rebuild the library by typing:

make all-target-winsup

Note that minimally, the top-level tree needs to contain:

Makefile.in, config, config-ml.in, config.guess, config.sub, configure, configure.in, include, install-sh, libiberty, ltconfig, ltmain.sh, missing, move-if-change, newlib, symlink-tree, winsup, ylwrap.

So another method that might work would be to start with a newlib and winsup snapshot and copy the missing pieces from the last major release source tree. You may also want to fix the EXE_LDFLAGS variable in winsup/Makefile.in so that winsup/utils/ programs link with the correct libcygwin.a.

Before you get started

First, note that our team only maintains the winsup directory (but can also make changes to newlib as well). That said, feel free to send us Cygwin-related patches for other tools -- we'll probably know what you should do with them if we're not in the position of making the change directly.

If your change is going to be a significant one in terms of the size of your code changes, be aware that you will have to sign over the copyright ownership of your code changes to Cygnus or the FSF (depending on the source file) before we can include your changes in the main source tree. Your employer may also have to send us a disclaimer stating that they have no claims to your contribution. This is necessary for liability reasons. Here is our standard assignment form for changes to Cygwin that you can fill out, sign, and send back. It shouldn't end up being that much of a pain. If you have any questions, please send them to cygwin@cygwin.com.

Finally, before you start modifying anything for real, you should probably join the cygwin-developers mailing list and send it mail describing what you want to do and how you want to do it so you don't waste time working on something we can't integrate. This also allows us to keep track of what's being worked on so efforts aren't duplicated.

Once you've finished making and testing your changes, you should send a context diff to the cygwin-patches list. As an example, if you modified winsup/syscalls.cc and have the original source file around as /winsup/syscalls.cc-orig, you would do the following in bash: 

      cd /c/src/winsup 

      diff -up syscalls.cc-orig syscalls.cc > syscalls.cc-patch 2>&1

Then send it to the cygwin-patches list in an email along with an explanation of what the change does. This must include a ChangeLog entry (model yours after other entries in the file winsup/ChangeLog). Then we'll test it and get back to you as soon as we can.

Note that you must be subscribed to cygwin-developers and cygwin-patches to send email to those lists. If you have a small patch for Cygwin and don't intend to contribute to ongoing development, it is not necessary to send your patch to the cygwin-patches mailing list. Instead, you may try your luck on the general cygwin mailing list. The responsible parties at Cygnus read the cygwin mailing list so they will eventually see your proposed change.

Please feel free to ask me any questions about any of this. I would love it if more people made useful contributions -- this can be one of the big advantages of free software; we all benefit from everyone else's work as well as our own...

You can get a good idea of the current bugs and problem areas by browsing the cygwin mailing list archives. In addition, check out the API calls portion of the FAQ and look for references to unimplemented functions. Or send mail to cygwin-developers@cygwin.com and see what people suggest. Please be advised that this list is intended only for people who are serious about augmenting the cygwin DLL. It is not intended as a vehicle for getting your general cygwin questions answered or for announcing ports of any associated Cygwin utilities. You will first have to join the list before you are able to send email.
